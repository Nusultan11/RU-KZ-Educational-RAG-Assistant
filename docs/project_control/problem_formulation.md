# Формулировка проекта

## Этап

Stage 1: Project Control + Dataset Inventory + EDA Plan.

Этот этап только планировочный. На нем нельзя реализовывать preprocessing, indexing, retrieval, generation, API, UI или QLoRA.

## 1. Цель проекта

Создать двуязычного образовательного RAG-ассистента для русского и казахского языков.

Ассистент должен:

- отвечать на вопросы пользователя на русском и казахском языках;
- использовать внешнюю базу знаний;
- показывать источники ответа;
- быть устойчивым к плохим или неполным вопросам;
- быть пригодным для дальнейшего production-deploy;
- развиваться через измеримый ML/LLM pipeline.

## 2. Проблема проекта

Обычная LLM может:

- выдумывать факты;
- отвечать без источников;
- плохо работать с локальными и узкими знаниями;
- нестабильно переключаться между русским и казахским языками;
- не объяснять, откуда взят ответ.

Для образовательного ассистента это критично: пользователь должен получать проверяемый ответ, основанный на найденных источниках, а не на догадке модели.

## 3. ML-задача

Построить измеримую систему поиска и генерации:

- найти релевантный контекст для вопроса;
- сгенерировать ответ на основе найденного контекста;
- сохранить язык вопроса, если это возможно;
- вернуть источники;
- измерять качество retrieval и generation фиксированными метриками.

## 4. RAG-задача

Retriever должен находить полезные документы или контексты для RU/KZ вопросов.

Generator должен:

- отвечать только по найденному контексту для фактических вопросов;
- добавлять источники;
- не выдумывать ответ, если контекста недостаточно;
- использовать safe fallback для случаев без релевантного контекста.

## 5. Data-задача

До preprocessing нужно понять структуру и качество данных.

Data-задача Stage 1:

- разделить retrieval-датасеты и instruction-датасеты;
- зафиксировать роль каждого датасета;
- определить, какие поля нужно найти в EDA;
- определить, какие проверки должны пройти данные;
- подготовить основу для будущего data quality report.

## 6. EDA-план

EDA должен ответить на вопросы:

- какие колонки есть в каждом датасете;
- где находятся question, answer, context, document, instruction, input, output и metadata;
- есть ли splits: train, validation, test;
- сколько записей, пустых значений и дублей;
- какие распределения длин у документов, вопросов, ответов и контекстов;
- какие языки реально есть в данных;
- есть ли HTML, спецсимволы, битые строки и нерелевантные пары;
- можно ли считать Recall@k, MRR и nDCG;
- пригодны ли instruction-данные для будущего SFT/QLoRA.

Подробный EDA-план находится в `docs/project_control/eda_plan.md`.

## 7. Критерии успеха проекта

Проект считается двигающимся в правильном направлении, если:

- ответы содержат источники;
- ответ соответствует языку вопроса;
- retrieval измеряется через Recall@k, MRR и nDCG;
- RAG измеряется через faithfulness, answer relevance, source correctness и language consistency;
- hallucinations уменьшаются относительно baseline;
- pipeline воспроизводится через конфиги и отчеты;
- preprocessing начинается только после EDA;
- QLoRA начинается только после baseline RAG, evaluation и error analysis.
